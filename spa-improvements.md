# SPA 개선 제안서

현재 구조를 기준으로, 바닐라 JS / CSS / HTML만으로 만든 SPA에서 특히 효과가 큰 개선 포인트를 정리했습니다.

## 1. 라우팅 구조를 조금 더 안전하게 만들기

현재 라우팅은 해시 기반으로 단순하고 이해하기 쉽지만, [javascript/route.js](javascript/route.js), [javascript/render.js](javascript/render.js) 조합에서는 404 처리와 상태 분리가 아직 단순합니다.

- 현재 경로가 없을 때 보여주는 404 화면을 별도 페이지로 분리하면 유지보수가 쉬워집니다.
- 라우트별 공통 레이아웃과 페이지 내용을 분리하면 페이지가 늘어나도 코드가 덜 복잡해집니다.
- 해시 변경만으로 이동하는 방식은 괜찮지만, 나중에 페이지가 많아지면 라우트 정의를 한 곳에서 관리하는 구조가 더 안정적입니다.

## 2. 렌더링과 비동기 데이터의 로딩 상태를 분리하기

[javascript/render.js](javascript/render.js)와 [javascript/api/weather.js](javascript/api/weather.js)를 보면, 화면 렌더링과 날씨 데이터 요청이 서로 느슨하게 연결되어 있습니다.

- 날씨 데이터를 가져오는 동안 보여줄 로딩 상태가 있으면 첫 화면 경험이 더 좋아집니다.
- API 실패 시 단순히 N/A를 넣는 것보다, 카드 영역에 실패 안내를 표시하면 사용자 입장에서 원인을 이해하기 쉽습니다.
- 렌더 함수가 화면 갱신만 담당하고, 데이터 조회는 별도 계층으로 두면 유지보수가 편해집니다.

## 3. i18n 상태 관리 일관성 점검하기

현재 [javascript/i18n.js](javascript/i18n.js)에는 언어 상태를 바꾸는 함수가 있지만, 변수명 일관성이 완전하지 않고 언어 전환 후 다시 렌더링되는 흐름도 더 명확하게 정리할 수 있습니다.

- currentLanguage와 currentLang 같은 상태 변수는 한 이름으로 통일하는 편이 안전합니다.
- 현재 언어를 기준으로 렌더링하는 구조를 명시적으로 만들면, 나중에 언어가 추가되어도 확장하기 쉽습니다.
- 문서 문자열이 화면에 직접 들어가는 부분은 키 기반으로 통일하면 번역 유지보수가 쉬워집니다.

## 4. 접근성을 보강하기

네비게이션 버튼과 프로필 카드 중심의 구조에서는 시각적인 완성도만큼 접근성도 중요합니다. [index.html](index.html)과 [javascript/render.js](javascript/render.js), 그리고 네비게이션 관련 CSS를 함께 점검하는 것이 좋습니다.

- 현재 활성 탭은 시각적으로만 구분되므로 aria-current 같은 의미 정보를 추가하면 좋습니다.
- 버튼과 링크의 역할이 섞이지 않도록, 실제 이동 요소는 의미에 맞는 태그를 유지하는 것이 좋습니다.
- 이미지에는 이미 alt가 있지만, 프로필 이미지와 장식용 이미지를 구분해서 더 정확한 대체 텍스트를 주면 좋습니다.
- 기본 문서 언어가 한국어라면 html 태그의 lang 값도 실제 사용 언어에 맞추는 것이 좋습니다.

## 5. 콘솔 로그와 디버깅 코드를 정리하기

현재 [javascript/render.js](javascript/render.js), [javascript/route.js](javascript/route.js), [javascript/api/weather.js](javascript/api/weather.js)에는 console.log가 남아 있습니다.

- 개발 중에는 유용하지만, 배포 전에 제거하거나 개발 모드에서만 남기는 방식이 좋습니다.
- 로그가 많아지면 실제 오류를 찾기 어려워질 수 있으므로, 에러 로그와 디버그 로그를 구분하는 편이 좋습니다.

## 6. CSS 구조를 컴포넌트 단위로 더 나누기

현재 CSS 폴더 구조는 이미 어느 정도 분리되어 있지만, 페이지가 더 늘어나면 스타일 관리 난도가 빠르게 올라갑니다.

- 공통 토큰은 root.css에 계속 두고, 레이아웃과 컴포넌트 스타일은 더 명확하게 분리하는 편이 좋습니다.
- 애니메이션과 기본 레이아웃, 페이지 전용 스타일을 섞지 않으면 수정 영향 범위를 줄일 수 있습니다.
- 반복되는 카드 스타일은 공통 클래스로 묶으면 유지보수가 쉬워집니다.

## 7. 확장성을 위한 추천 우선순위

1. 언어 상태 관리 정리와 렌더 흐름 통일
2. 라우팅 404 및 페이지별 구조 분리
3. 날씨 API 로딩과 실패 상태 추가
4. 접근성 속성 보강
5. 디버그 로그 제거 및 배포 정리

## 짧은 결론

지금 구조는 SPA의 핵심은 잘 잡혀 있습니다. 다만 페이지 수가 늘어나거나 i18n, 비동기 데이터, 접근성이 조금만 복잡해져도 관리 비용이 커질 수 있으니, 지금 단계에서 상태 관리와 라우팅, 로딩 상태를 먼저 다듬는 것이 가장 효율적입니다.
